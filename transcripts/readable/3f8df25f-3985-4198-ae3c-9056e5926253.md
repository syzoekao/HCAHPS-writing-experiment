Good — that combination is a sensible sweet spot. You're not trying to make a single up-or-down claim about Claude's writing ability; you're trying to *learn* what works while still producing a real manuscript. The audit trail is what lets the experience itself become the data.

Here's what I'd propose setting up before we start writing. It's deliberately minimal — just enough scaffolding that nothing gets lost, without over-engineering a protocol you'll outgrow.

**Proposed setup**

1. **Git repo** — `git init` here. Three conventions:
   - Commit message prefix marks the stage: `outline:`, `draft:`, `self-review:`, `peer-review:`, `revise:`, `meta:` (for protocol/log changes).
   - Each commit message names the agent or skill that produced the change (`drafter agent`, `manual edit`, etc.) so `git log` doubles as a provenance record.
   - Tag at handoffs: `v1-outline`, `v1-draft-complete`, `v1-after-peer-review`, `v2-revised`. Tags let you `git diff v1-draft-complete v2-revised` to see exactly what one revision cycle changed.

2. **Two top-level documents** (small, written by hand, no LLM):
   - `experiment-readme.md` — one page: what this experiment is, the scope (which sections of the HCAHPS paper, what success looks like to *you*), audit-trail conventions, model versions in use.
   - `experiment-log.md` — running journal. Dated entries. For each session: goal, what I asked Claude to do, what happened, what surprised me, what I'd change. This is the *interpretive* layer that raw transcripts can't give. ~5 minutes of journaling per session, end-of-day.

3. **Transcript export** — Claude Code already saves every session as JSONL at `~/.claude/projects/-workspaces-HCAHPS/`. A small script (`scripts/export-transcript.sh`) that:
   - Copies new JSONL files into `transcripts/raw/`
   - Generates a readable markdown rendering in `transcripts/readable/` (user prompts + assistant prose + tool-call summaries)
   - Runs end-of-day or after each major milestone, then committed with `meta: export transcripts`.

4. **Artifact provenance, lightweight** — don't bother with frontmatter on every file. `git blame` + commit messages give you enough. The exception: when a file has been touched by both Claude and you, a one-line note at the top of the log entry covering that session is enough.

**What we're explicitly not doing yet**

- No formal evaluation rubric. Once you have one or two runs, you'll know better what dimensions matter (and which ones the skill-based workflow already enforces, like word count and citation integrity).
- No multiple trials, no controls, no statistics. This is n=1 with rich qualitative documentation.
- No transcript redaction tooling. Your transcripts will go into a private repo (or stay local); revisit if/when you want to share.

**Minor practical notes**

- `.gitignore` should exclude `*.dta`, `*.csv`, the `OneDrive` data paths, and any generated `*.docx` (regenerable from `.md`). Manuscripts and transcripts get tracked; data does not.
- The Claude Code project path uses dashes in place of slashes: `~/.claude/projects/-workspaces-HCAHPS/`. The export script needs to know that.
- One thing worth deciding now: do you want transcripts committed to the same repo as the manuscript, or kept in a sibling repo? Same repo is simpler; sibling repo keeps the manuscript history clean if you ever want to share the writing without the experiment exhaust.
